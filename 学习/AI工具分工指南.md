# AI 工具分工指南 — 什么问题问谁？

> 基于 LuckCup 项目的真实踩坑经历总结。
> 最后更新：2026-03-22

---

## 一、核心判断方法

问自己一个问题：**这个问题的答案会不会随着时间变化？**

| 答案 | 找谁 | 为什么 |
|------|------|--------|
| 不会变 | Claude（本地 AI） | 编程语言语法、算法逻辑、代码架构——这些 10 年前和 10 年后都一样 |
| 会变 | Kimi / GPT 联网搜索 | 云平台产品、价格、配置界面、API 是否下线——这些随时在变 |

---

## 二、具体分工表

### Claude 擅长的（不需要联网）

| 场景 | 举例 |
|------|------|
| **读代码、改代码** | "帮我在收入页加个删除按钮" |
| **分析报错信息** | "这个 500 错误是什么原因" |
| **设计方案** | "后台补录功能应该怎么设计" |
| **解释技术概念** | "HashRouter 和 BrowserRouter 的区别" |
| **写 SQL** | "帮我生成历史数据导入的 SQL" |
| **Git 操作** | "帮我提交代码、整理分支" |
| **对比分析** | "这两个方案哪个更适合我" |
| **代码审查** | "上次的改动有没有问题" |

### 需要联网搜索的（问 Kimi / GPT 联网 / 自己查文档）

| 场景 | 举例 | 今天踩的坑 |
|------|------|-----------|
| **云平台的具体操作步骤** | "腾讯云 COS 怎么绑定自定义域名" | Claude 不知道 COS 有"默认源站"和"静态网站源站"两种类型 |
| **产品是否还在线** | "API 网关还能用吗" | Claude 不知道 API 网关已停售，你自己发现的 |
| **控制台里的选项含义** | "源站类型选哪个" | Claude 猜不到控制台里有这个下拉选项 |
| **平台的最新限制** | "2024年后创建的桶有什么限制" | COS 默认域名不能浏览器预览，这是 2024 年新政策 |
| **DNS/域名配置** | "CNAME 记录怎么填" | Claude 能告诉你该指向哪，但具体操作步骤要看平台文档 |
| **费用和计价** | "CDN 一个月多少钱" | 价格随时在变 |
| **第三方服务的当前状态** | "这个 API 还能调吗" | 服务可能已下线或改版 |

## 三、怎么判断"该去问了"

你不需要事先知道答案，只需要认出这些**信号**：

### 信号 1：Claude 给的方案涉及云平台控制台操作

Claude 说："你去 COS 控制台改一下 XXX"
→ **去问 Kimi**："腾讯云 COS 控制台怎么改 XXX，具体步骤是什么"

因为 Claude 知道原理但不知道当前控制台长什么样、有哪些选项。

### 信号 2：Claude 给了两个方案但不确定哪个能用

Claude 说："要么用 API 网关，要么用函数 URL"
→ **去问 Kimi**："腾讯云 SCF 的 API 网关还能用吗？函数 URL 和自定义域名怎么选？"

因为产品线可能已经变了。

### 信号 3：你在控制台看到了 Claude 没提到的选项或提示

你看到控制台写着"API 网关触发器功能已下线"
→ 这就是 Claude 不知道的新信息，**直接告诉 Claude**，它会调整方案

你看到源站类型有个下拉框，选项是"默认源站/静态网站源站"
→ **去问 Kimi**："这两个有什么区别，我应该选哪个"

### 信号 4：同一个问题反复排查还是不通

今天的例子：DNS 改了，COS 也查了，防盗链也没开，但就是 403
→ **该去问了**。Claude 能排查的都排查了，剩下的是平台配置层面的问题

### 信号 5：涉及"最新""当前""现在"的问题

"现在腾讯云推荐用什么方案部署前端？"
→ **直接问 Kimi**，不要问 Claude。Claude 的知识可能已经过时。

## 四、今天三个工具各自的贡献

用今天的部署排障做一个完整的例子：

```
第一步（Claude）
  你：SCF 的 API 网关地址在哪？
  Claude：应该在 API 网关控制台里...
  → 这一步 Claude 答错了，因为 API 网关已经下线

第二步（你自己）
  你在腾讯云控制台发现：API 网关已停售
  你把这个信息告诉了 Claude
  → 这一步是你自己发现的，Claude 和 Kimi 都没帮到

第三步（Claude）
  Claude 根据你的信息调整：那就用 SCF 函数 URL
  你测试 /health 通了
  你决定绑定自定义域名 api.lcup.cn
  → Claude 做了正确的逻辑调整

第四步（Claude）
  前端部署后 lcup.cn 返回 AccessDenied
  Claude 排查了：文件在、权限对、防盗链没开
  Claude 判断是 DNS 端点问题，建议改 CNAME
  → 方向对了，但不够深入

第五步（Claude 的局限）
  DNS 改了还是 403
  Claude 给了两个方案：开 CDN 或用 /index.html 路径
  → 到这里 Claude 已经到达了能力边界

第六步（Kimi）
  你拿着"为什么必须开 CDN"去问 Kimi
  Kimi 联网查到：COS 自定义域名有"默认源站"和"静态网站源站"两种类型
  → 这就是 Claude 不知道的平台细节

第七步（你自己）
  你在 COS 控制台把源站类型改成"静态网站源站"
  问题解决
  → 最终执行是你自己完成的
```

## 五、实用口诀

> **代码问 Claude，平台问 Kimi，控制台靠自己。**

- **Claude** = 你的程序员搭档（写代码、分析逻辑、设计方案）
- **Kimi** = 你的搜索助手（查文档、查最新信息、查平台配置）
- **你自己** = 项目老板（最终决策、控制台操作、发现 AI 不知道的现场信息）

三个角色缺一不可。Claude 不能替代 Kimi 查网页，Kimi 不能替代 Claude 写代码，AI 都不能替代你在控制台上看到的实际情况。

---

## 六、什么时候三个一起用

最高效的工作方式是**并行**：

1. 让 Claude 改代码、构建打包
2. 同时问 Kimi 部署步骤和注意事项
3. 自己在控制台准备好环境

这样 Claude 改完代码的时候，你已经知道怎么部署了，直接上传就行。

今天的教训是串行做的：Claude 改完 → 上传 → 报错 → 排查 → 问 Kimi → 再改。如果一开始就同时问 Kimi "COS 静态网站自定义域名有什么注意事项"，可能就不会踩这个坑。
