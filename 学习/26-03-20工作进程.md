## 今日问题链路

### 1. 今天实际遇到的问题

- 历史导入时报 `ERROR 1062 Duplicate entry ... for key 'expenses.PRIMARY'`
- 腾讯云 DMS 执行长 SQL 时出现 `null[502]`
- 已注册手机号无法登录，注册也失败
- 登录后页面没有历史数据
- 前端重新部署后，页面与数据表现反复不一致
- 最终定位到前端请求落到了 `https://lcup.cn/api/...`，返回 `405`

### 2. 已确认的根因

- `expenses / backend_entries` 的导入 SQL 最初不是幂等写法，重跑时容易撞主键
- 腾讯云 DMS 的 `502` 更像执行通道异常，不等于整批 SQL 都失败
- 线上 SCF 一度缺少 `JWT_SECRET`，导致 `/auth/login` 返回 `500`
- 当前可登录账号的 `shop_id` 与历史导入数据的 `shop_id` 不一致
- 生产环境前端 API 地址没有被显式固定，触发了 `api.js` 中的 fallback：
  - `const BASE_URL = import.meta.env.VITE_API_URL || '/api';`
- `'/api'` 只在本地 Vite 开发代理里成立，不适用于 COS 生产环境

### 3. 今天额外暴露出的架构偏移

- 文档长期记录的线上地址是 `https://lcup.cn/#/income`
- 但 git 初始提交中的 `luckcup-web/src/main.jsx` 实际使用的是 `BrowserRouter`
- 仓库中也没有保留生产打包时使用的 `.env.production` 或等效配置
- 这说明“第一版线上实际部署方式”和“当前仓库里的最早提交代码”本来就存在偏差

- 以后只要涉及部署架构、路由模式、API 地址来源、域名代理、COS 与 SCF 职责边界

## 关键经验

- 腾讯云 DMS 跑大批量 SQL 时，优先使用幂等写法，避免中断后无法安全重跑
- 看到 `502` 时，先确认具体 SQL 是否已经落库，不要立刻重复整批执行
- 登录接口返回 `500` 时，应先查环境变量和 SCF 日志，而不是只盯着账号密码
- 登录成功但页面没数据时，优先检查账号对应的 `shop_id` 是否和历史数据一致
- 生产环境里 `'/api'` fallback 不能替代真实部署配置，必须显式固定 `VITE_API_URL`
- 路由模式、API 地址来源、COS/SCF 职责边界都属于部署契约，必须先确认后改动
