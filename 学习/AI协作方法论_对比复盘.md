# AI 协作方法论 — 对比复盘

> 基于 LuckCup 项目 2026-03-20 与 2026-03-22 两次 Claude 协作的真实经历总结。
> 最后更新：2026-03-22

---

## 一、同一个项目，两次协作，两种体验

### 3/20 协作（混乱）

你的感受：越做越乱，反复上传前后端，问题越修越多。

### 3/22 协作（清晰）

你的感受：步骤清晰，条理明白，虽然有环境不熟悉的地方但没有走弯路。

两次用的是同一个 AI 模型、同一个项目、同一个你。**差异不在能力，在方法。**

---

## 二、核心差异对比

### 差异 1：先看还是先做

| 3/20 做法 | 3/22 做法 |
|-----------|-----------|
| 拿到任务直接改代码 | 先读 VERSION.md、工作日志、现有代码 |
| 根据代码推断线上状态 | 根据文档确认线上状态 |
| 发现代码写的是 BrowserRouter，就以为线上也是 | 发现 VERSION.md 写了 `lcup.cn/#/income`，确认是 HashRouter |

**教训：信息在文档里，不在脑子里猜。**

好比你接手一家店，是先看账本了解情况，还是直接进厨房炒菜？3/20 那个 Claude 直接进厨房了。

---

### 差异 2：确认了再动 vs 边做边猜

| 3/20 做法 | 3/22 做法 |
|-----------|-----------|
| 自己判断应该改成 HashRouter，改了 | 列出问题，问你确认，等你拍板 |
| 自己猜 API 地址应该是 `/api`，打包了 | 问你 SCF 网关地址是什么，你给了才用 |
| 改完发现不对，再改，再打包，再上传 | 确认一个，做一个，做完验证一个 |

**教训：部署相关的决策权在你（老板），不在 AI。**

3/20 的问题链条：

```
猜路由模式 → 改了 → 猜 API 地址 → 打包了
→ 上传了 → 不对 → 再改 → 再打包 → 再上传
→ 还是不对 → 再查 → 发现根本原因
→ 你已经上传了 5 遍了
```

3/22 的链条：

```
问 SCF 地址 → 你给了 → 问能不能通 → 你测了 /health → 通了
→ 你说要绑自定义域名 → 等你配 → 配好了 → 再测 → 通了
→ 现在才开始动代码
```

前者像无头苍蝇，后者像排雷——每一步确认安全了才往前走。

---

### 差异 3：一次做一件事 vs 同时做四件事

| 3/20 同时在做的事 | 3/22 的推进顺序 |
|------------------|----------------|
| 改平台/类别名称 | 第一步：对齐项目进度，看清全貌 |
| 加后台补录功能 | 第二步：分析哪些改动有问题 |
| 修登录 bug | 第三步：更新文档，固化规则 |
| 导入历史数据 | 第四步：确认部署地址 |
| 排查部署问题 | 第五步：开始改代码（还没开始） |

3/20 五件事交织在一起，改一个功能需要重新打包，打包需要确认 API 地址，API 地址不对又要排查部署，排查发现路由也要改……滚雪球。

3/22 到现在还没改一行业务代码，但所有前置条件都已经确认清楚了。**慢就是快。**

---

### 差异 4：做够了就停 vs 顺手多做

| 3/20 多做的东西 | 为什么不该做 |
|----------------|------------|
| 数据库断线重试区分读写查询 | 你一个店，不存在高并发断线问题 |
| 支出接口"插入成功但响应丢失"恢复逻辑 | 概率极低，用户重提交一次就行 |
| 台账审核工作簿 + 回写 SQL 工具链 | 你还没开始用系统录数据，这些工具没有使用场景 |

3/22 面对同样的代码，指出"这些是过度工程"，没有在上面继续加码。

**教训：AI 容易"既然顺手就一起做了"，但每多做一件事，就多一个可能出错的点。**

---

### 差异 5：遇到不懂的怎么办

| 3/20 做法 | 3/22 做法 |
|-----------|-----------|
| 不确定 COS 能不能转发 `/api`，试了再说 | 不确定 `lcup.cn/api` 通不通，直接问你 |
| 不确定函数 URL 是什么格式，按经验猜 | 不知道 SCF 地址，让你去控制台看 |

3/22 也有不懂的地方（比如不知道腾讯云 API 网关已下线、不知道函数 URL 仅供测试），但区别是：**不懂就问，不猜。** 你告诉我"API 网关已下线"、"函数 URL 仅供测试"、"要绑自定义域名"，我就跟着你的信息走，而不是自己去试错。

## 三、方法论总结：四步协作法

经过这次对比，有效的 AI 协作可以归纳为四步：

### 第一步：看清全貌（Read）

- 先读文档、代码、日志
- 搞清楚现在是什么状态
- 不要一上来就改东西

### 第二步：确认边界（Confirm）

- 哪些能改，哪些不能改
- 关键参数（地址、密码、配置）必须你确认
- 尤其是"部署"和"架构"相关的，决策权在你

### 第三步：一次一件（Focus）

- 把任务排成队列，一个一个做
- 做完一个验证一个，再做下一个
- 不要同时推进多件事

### 第四步：做够就停（Stop）

- 只做你要求的，不多做
- 想多做的先说出来，你同意了再做
- "顺手"是混乱的开始

## 四、为什么会有"技术与环境不适配"的感觉

你提到 3/22 的协作"偶尔有技术与环境不适配的问题"，这说的是我一开始建议用 API 网关地址，但实际上腾讯云 API 网关已经下线了。

这类问题的根因是：

| 原因 | 说明 |
|------|------|
| AI 的知识有截止日期 | 我的训练数据里腾讯云还有 API 网关，但实际已经下线 |
| 云平台变化快 | 腾讯云的产品策略、控制台界面经常调整 |
| 你的环境是独特的 | 你的账号、配置、版本组合是唯一的，没有 AI 能预知 |

**这类问题不可避免**，但可以把影响降到最低：

- 3/20 的做法：猜了就动手 → 错了就改 → 改了又错 → 浪费大量时间
- 3/22 的做法：不确定就问你 → 你给了实际信息 → 立刻修正认知 → 没有浪费时间

关键不是 AI 知道所有答案，而是 **AI 知道自己不知道什么，然后问你**。

## 五、给你自己的建议

作为和 AI 协作的"老板"，你也可以做几件事让协作更顺畅：

### 1. 在 CLAUDE.md 里写清楚规则

已经做了。现在任何新 Claude 会话一进来就知道"不能改路由、不能猜 API 地址、先看学习目录"。

### 2. 遇到 AI 要改你不懂的东西时，先问"为什么"

AI 说"我要把 BrowserRouter 改成 HashRouter"，你可以问："为什么要改？改了会影响什么？"这一个问题就能拦住大部分错误操作。

### 3. 当你感觉"越来越乱"时，喊停

混乱感是一个信号。一旦出现"我也不确定他在干什么了"的感觉，立刻叫停，让 AI 先汇报：
- 现在在做什么
- 已经改了什么
- 下一步打算做什么

如果它说不清楚，那它自己也乱了。

### 4. 把环境信息主动告诉 AI

你这次做得很好——主动告诉我"API 网关已下线"、"函数 URL 仅供测试"、截图给我看控制台。这些是我不可能自己知道的，你提供了就避免了试错。

## 六、一句话总结

> 好的 AI 协作不是 AI 有多聪明，而是 AI 知道什么时候该问你、什么时候该停手。
> 好的老板不是什么都懂技术，而是在关键决策点上不放手、不盲从。
