# LuckCup 运营系统 — 技术名词学习手册

> 用大白话理解技术，用生活例子记住概念。随项目进展持续更新。

---

## 1. 前端（Frontend）—— 用户看到的界面

就像餐厅的门面、装修和菜单。

| 名词 | 一句话解释 | 生活类比 |
|------|-----------|---------|
| **React** | 搭建界面的工具箱，全球最流行 | 乐高积木，拼出各种页面 |
| **Tailwind CSS** | 给界面"化妆"的工具 | 装修时的配色和材料方案 |
| **PWA** | 让网页伪装成APP | 添加到主屏幕后像微信一样有图标 |
| **组件（Component）** | 可重复使用的页面零件 | 乐高的单个零件，到处能用 |
| **响应式设计** | 页面自动适应不同屏幕 | 折叠桌，空间大就展开，小就折起 |

---

## 2. 后端（Backend）—— 处理数据的"厨房"

用户看不到，但所有数据处理都在这里完成。

| 名词 | 一句话解释 | 生活类比 |
|------|-----------|---------|
| **Node.js** | 运行后端程序的平台 | 厨房的灶台 |
| **Express** | 后端的工作流程框架 | 厨房的操作流程表 |
| **API** | 前端和后端之间的传输窗口 | 传菜窗口 |
| **JSON** | 前后端交换数据的格式 | 一张写清楚内容的单子 |

---

## 3. 数据库（Database）—— 存放数据的"仓库"

所有收入、支出、设置信息都存在这里。

| 名词 | 一句话解释 | 生活类比 |
|------|-----------|---------|
| **MySQL** | 当前项目使用的数据库（腾讯云版） | 电子账本，几百万笔记录秒查 |
| **PostgreSQL** | 另一种开源数据库，Vercel 时期用过 | 同上，只是品牌不同 |
| **表（Table）** | 存一类数据的单元 | Excel的一个工作表 |
| **字段（Column）** | 表里的"列" | 你Numbers清单里的"日期""金额"等列 |
| **id（主键）** | 每条数据的唯一编号 | 身份证号 |
| **shop_id（外键）** | 数据之间的关联线 | 快递单上的收件人电话 |
| **数据迁移** | 修改数据库结构 | Excel里新加一列 |

## 4. 部署与运维 —— "把店开在哪条街"

代码写好后放到网上让人访问。

| 名词 | 一句话解释 | 生活类比 |
|------|-----------|---------|
| **腾讯云 COS** | 当前使用的前端文件托管 | 把装修好的门面放在商场里展示 |
| **腾讯云 SCF** | 当前使用的后端运行平台（Serverless） | 按需租厨房，有人点菜才开火，不点不收钱 |
| **API 网关** | SCF 前面的"门卫"，把外部请求转发给 SCF | 商场前台，客人说要找哪家店，前台指路 |
| **Vercel** | 之前用的前端托管，国内访问不了已弃用 | 国外免费商铺，可惜国内客人进不去 |
| **Railway / Render** | 之前用的后端托管，已弃用 | 国外租的厨房 |
| **域名** | 网站地址，如 lcup.cn | 店铺的门牌号 |
| **VITE_API_URL** | 前端构建时写死的后端地址 | 菜单上印的电话号码，印错了客人就打不通 |
| **HashRouter** | 前端路由模式，URL 里带 `#` 号 | 一栋楼只有一个大门（index.html），`#` 后面是房间号 |
| **BrowserRouter** | 另一种路由模式，URL 不带 `#` | 每个房间都有独立大门，但需要物业配合（服务器配置） |
| **环境变量** | 不写在代码里的配置信息 | 店长脑子里记的密码，不贴在墙上 |

## 5. 安全与登录 —— "门禁和钥匙"

| 名词 | 一句话解释 | 生活类比 |
|------|-----------|---------|
| **Token（令牌）** | 登录后的通行证，支持多设备同时在线 | 游乐园手环，老板店长各一个 |
| **数据隔离** | 各加盟商数据互不可见 | 银行保险柜，各人各的 |
| **认证** | 确认"你是谁" | 保安查门禁卡 |
| **授权** | 确认"你能做什么" | 能进小区但只能开自己家的门 |
| **HTTPS** | 数据加密传输 | 用保险柜寄快递，截获了也打不开 |

| 名词 | 一句话解释 | 生活类比 |

## 7. 项目实战经验 —— "踩过的坑和学到的本事"

> 这些不是课本知识，是真金白银的项目经历换来的。每一条都对应一个真实发生的问题。

### 7.1 仓库管理经验

**经验：提交到 GitHub 的代码必须和线上跑的版本一致**

- 我们踩的坑：第一次提交代码时，代码里写的是 `BrowserRouter`，但实际部署到腾讯云时已经改成了 `HashRouter`。结果后面的 Claude 看到代码以为线上是 BrowserRouter，改来改去浪费了大量时间。
- 教训：每次部署上线后，把改过的代码提交一次。代码仓库就是"正式版存档"，不能让它和线上版本脱节。
- 生活类比：就像你店里的价目表改了，但总部系统没更新，下次盘点就对不上。

**经验：项目文件要分"给人看的"和"给机器看的"**

- `学习/` 目录里的文档 → 给你自己看的，帮你理解项目
- `CLAUDE.md` → 给 Claude 看的，告诉它项目规则和约束
- 代码文件 → 给机器执行的
- 踩的坑：之前 `CLAUDE.md` 里只有工作日志，没有写清楚部署规则，导致 Claude 不知道哪些东西不能随便改

### 7.2 部署经验

**经验：改部署相关的东西之前，先确认"基线"**

- 部署基线 = 当前线上到底是怎么跑的（用什么路由、API 地址从哪来、前端放在哪、后端放在哪）
- 我们踩的坑：Claude 没确认基线就同时改了路由模式 + 重新打包，结果 API 地址也错了，前端请求到了错误的地方（COS 返回 405），导致反复上传前后端包排查
- 教训：涉及路由模式、API 地址、COS/SCF 职责的改动，必须先问用户确认，改完一个验证一个，不要同时改多个
- 生活类比：装修房子时，不能同时拆墙、改水电、换地板。先确认承重墙在哪，再一步步来

**经验：`VITE_API_URL` 是前端的"命根子"**

- 前端构建（`npm run build`）时，`VITE_API_URL` 会被写死到打包产物里
- 如果构建时没传这个变量，代码会 fallback 到 `'/api'`，这在本地开发没问题（Vite 会代理），但放到 COS 上就废了——COS 不会帮你转发 `/api` 请求
- 教训：每次打包前端，必须明确传入 `VITE_API_URL=你的SCF网关地址`

**经验：COS 和 SCF 是两个独立的东西，别搞混**

- COS（对象存储）= 只放文件，就像一个网盘，用户访问 `lcup.cn` 时拿到 HTML/JS/CSS
- SCF（云函数）= 跑后端代码，处理登录、存数据等请求
- 它们之间没有任何自动关联。前端要访问后端，必须通过 API 网关地址直接访问 SCF
- 踩的坑：以为 COS 会自动把 `/api` 开头的请求转发给 SCF，实际上不会

### 7.3 数据库经验

**经验：导入数据的 SQL 必须用 `INSERT IGNORE` 或 `ON DUPLICATE KEY UPDATE`**

- 我们踩的坑：第一版导入脚本用普通 `INSERT`，在腾讯云 DMS 执行到一半断了（502），重新执行时前面已经插入的数据报主键冲突，整批失败
- 教训：任何批量导入 SQL 都要设计成"可以安全重跑"的。用 `INSERT IGNORE`（重复就跳过）或 `ON DUPLICATE KEY UPDATE`（重复就更新）
- 生活类比：往账本上抄数据，如果抄到一半被打断，下次从头抄的时候，已经抄过的要跳过，不能报错说"这行已经有了"

**经验：`shop_id` 是数据归属的核心，必须对得上**

- 每条收入、支出数据都绑定一个 `shop_id`（店铺编号）
- 用户登录后，系统根据用户的 `shop_id` 筛选数据
- 我们踩的坑：历史数据是用一个 `shop_id` 导入的，但当前登录账号绑的是另一个 `shop_id`，导致"登录成功但看不到数据"
- 教训：导入数据之前，先确认目标 `shop_id` 是否和当前登录账号一致

**经验：腾讯云 DMS 执行大 SQL 时，看到 502 不要慌**

- 502 是 DMS 控制台的网络超时，不代表 SQL 没执行
- 正确做法：先查一下数据有没有落库（`SELECT COUNT(*) FROM 表名`），再决定要不要重跑

### 7.4 协作经验（和 AI 一起开发）

**经验：每次新开对话，Claude 不记得上次聊的内容**

- Claude 每次对话是独立的，它只能通过 `CLAUDE.md` 和项目文件了解上下文
- 所以 `CLAUDE.md` 写得好不好，直接决定下一个 Claude 会不会犯错
- 教训：`CLAUDE.md` 里必须写清楚"什么能改、什么不能改、部署规则是什么"，不能只写工作日志

**经验：不要让 Claude 同时干太多事**

- 我们踩的坑：3/20 那天同时在做功能开发（后台补录）、数据导入（历史/2月）、线上排障（登录/部署），三件事互相干扰
- 教训：一次只推进一个方向。先把线上跑通，再加新功能，再导数据。就像开店一样——先把门开了，再上新品，再盘库存

**经验：Claude 容易"过度工程"——做你没要求的事**

- 你说"修一下登录问题"，它可能顺手加了断线重试优化、响应丢失恢复、台账审核工具链
- 这些东西单独看都"有道理"，但在当前阶段完全不需要，反而增加了代码复杂度
- 教训：明确告诉 Claude "只做我说的，不要多做"。如果它要做额外的事，让它先说出来，你同意了再做

**经验：涉及"部署"和"架构"的改动，必须你来拍板**

- 代码逻辑的修改（比如修个 bug、加个接口）Claude 可以自己判断
- 但部署方式的改动（路由模式、API 地址、平台选择）必须你确认
- 因为这些改动会影响到你需要手动操作的部分（上传 COS、配置 SCF、设置环境变量）
